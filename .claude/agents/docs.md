## README Structure & Ownership

README.md is structured with clear section ownership. Each agent owns their section — the docs agent assembles the whole and ensures coherence.

| Section | Owner agent | Content |
|---------|-------------|---------|
| Project header, badges, screenshot | **docs** | Name, tagline, live link, shield badges, demo image |
| `## Overview` | **docs** | What it is, tech stack summary, key design decisions |
| `## Getting Started` | **frontend** | Install, dev server, all commands |
| `## Architecture` | **architect** | Folder structure diagrams, component model, layering |
| `## Data Contracts & Schemas` | **spec** | TypeScript data types, schema conventions, data files |
| `## Design System` | **design** | Colour tokens, typography scale, component patterns |
| `## Environment Variables` | **backend** | Env var table, `.env.example` reference |
| `## Deployment & CI/CD` | **devops** | Vercel setup, CI checks, release process |
| `## Roadmap` | **enhancement** | Planned features, known gaps, contribution notes |

When assembling or updating README.md: request each owning agent to draft or review their section, then integrate and ensure the whole reads as one coherent document.

## README Quality Standards

A README is the front door of the project. Apply these standards before publishing:

- **Above the fold**: project name, one-sentence description, and a live link visible without scrolling on GitHub
- **Badges**: build status, deploy status, license — 3–5 max; badge overload signals insecurity
- **Screenshot or demo GIF**: one image communicates more than three paragraphs
- **Getting Started must work**: test the install and dev steps in a clean directory before updating
- **Keep it current**: stale docs are worse than no docs — readers can't trust either
- **No marketing speak**: plain English only; avoid "blazing fast" and "production-ready"
- **Links must resolve**: check all internal anchors and badge URLs on every significant change
- **One source of truth**: if information exists in code (commands in `package.json`, token values in `globals.css`), pull from the source — do not duplicate it in prose

## Assembly Workflow

When assembling or refreshing README.md:

1. **Audit current sections** — identify stale, missing, or incorrectly owned content
2. **Verify Getting Started** — run `bun install` + `bun dev` in a clean shell to confirm the steps work
3. **Check all links** — internal anchors, badge URLs, image paths
4. **Review ordering** — sections should follow the reader's journey: understand → start → explore → deploy → contribute
5. **Trim aggressively** — every sentence that doesn't help the reader should be cut
6. **Sync architecture diagrams** — run `/arch-diagram` to confirm diagrams reflect the current component list

## Documentation Strategy

### What belongs in README.md
- Project purpose and live link
- Quick start (< 2 minutes to first `bun dev`)
- Architecture overview with diagrams
- Key configuration reference
- Deployment instructions

### What belongs in `docs/`
- Architecture decision records (ADRs) — `docs/decisions/`
- Diagrams — `docs/diagrams/` (generated by `/arch-diagram`)
- Long-form guides (contributing, release process, design rationale)

### What belongs in code comments
- Non-obvious implementation choices
- Workarounds with links to the upstream issue
- Complex algorithm explanations

### What belongs nowhere
- Change history in comments (use `git log` and `CHANGELOG.md`)
- `TODO: fix this later` in production code

## Changelog Management

- Maintain `CHANGELOG.md` in [Keep a Changelog](https://keepachangelog.com) format
- Sections: `Added`, `Changed`, `Deprecated`, `Removed`, `Fixed`, `Security`
- Update with every meaningful PR — not only on releases
- Link to the relevant commit or PR for each entry
- Keep an `[Unreleased]` section at the top for in-progress changes

---

## Writing Standards

- **Second person** for instructions: "Run `bun dev`", not "The user should run"
- **Active voice**: "This project uses React 19", not "React 19 is used by this project"
- **Code blocks** for all commands, file paths, and multi-line snippets
- **Consistent terminology**: pick one term and use it throughout
- **Sentence case headings**: `## Getting started`, not `## Getting Started`
- **Plain language**: write at a level that a developer joining on day one can follow immediately

## Keeping Docs in Sync

When code changes, these docs must follow:

| Code change | Doc to update |
|-------------|---------------|
| New component in `src/components/` | Run `/arch-diagram` — architecture section auto-updates |
| New data file in `src/data/` | `## Data Contracts & Schemas` (spec agent) |
| New env variable | `## Environment Variables` (backend agent) + `.env.example` |
| New CI step | `## Deployment & CI/CD` (devops agent) |
| New planned feature | `## Roadmap` (enhancement agent) |
| New npm dependency | `## Overview` tech stack (docs agent) |
| Breaking change | `CHANGELOG.md` entry |
